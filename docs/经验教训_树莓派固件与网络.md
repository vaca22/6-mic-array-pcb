# 经验教训：树莓派固件、网络与 6 麦测试

本文档记录在树莓派 4B + ReSpeaker 6-Mic 环境下的实操教训，避免重复踩坑。

---

## 一、固件与首次启动

### 1.1 固件 vs 系统镜像

- **固件包**（如 GitHub raspberrypi/firmware 的 ~3.7MB 压缩包）：仅含 boot/GPU 固件（`start*.elf`、`fixup*.dat` 等），用于在**已有系统**的卡上升级启动固件。
- **系统镜像**（如 Raspberry Pi OS Lite 的 ~500MB+ .img.xz）：完整操作系统，用于**空卡**首次刷写。
- 空 TF 卡必须刷**系统镜像**才能启动；只写固件包无法启动。

### 1.2 无显示器 (Lite) 首次登录

- 新版本 Raspberry Pi OS **默认不启用 SSH**，且**无默认用户 `pi`**，需在首次启动前在 **boot 分区**做两件事：
  1. **启用 SSH**：在 boot 分区创建空文件 `ssh`（无扩展名）。
  2. **创建用户**：在 boot 分区创建 `userconf.txt`，内容一行：`用户名:加密密码`。  
     加密密码生成：`echo '你的密码' | openssl passwd -6 -stdin`，取输出与用户名拼成 `用户名:哈希`。
- 若只做 `ssh` 不做 `userconf.txt`，SSH 会显示 “Permission denied” 且提示需先完成用户设置（无显示器时无法完成）。
- **建议**：刷完官方镜像后、**第一次上电前**，用本仓库的脚本或「SSH 直接可用固件」在 boot 分区写好 `ssh` + `userconf.txt`，再上电。

### 1.3 制作「SSH 直接能用」的固件

- 不改 rootfs，只改 **boot 分区**即可：在镜像的 boot 分区里预先放入 `ssh` 和 `userconf.txt`，再将该镜像写入 TF 卡，首次启动即可 SSH 登录。
- 仓库内提供 `scripts/make-ssh-ready-image.sh`：基于官方 Raspberry Pi OS Lite 镜像，解压 → 挂载 boot 分区 → 写入 `ssh` + `userconf.txt` → 卸载，得到新的 .img，再对该 .img 做 `dd` 即可得到「刷完即能 SSH」的固件。
- 若不想改镜像文件，也可在**刷好官方镜像、尚未上电**时，把卡插回电脑，用脚本往 boot 分区写 `ssh` + `userconf.txt`（与之前 `pi-headless-user.sh` 思路一致）。

## 二、网络与板子失联

### 2.1 板子突然连不上网

- **现象**：之前能 SSH（如 `pi@192.168.167.20`（IP 以项目根目录 `pi-ip.conf` 为准）），某次操作（如安装 seeed-voicecard、改配置、重启/断电）后无法 ping/SSH。
- **可能原因**（按优先级排查）：
  1. **DHCP 租约或路由器分配了新 IP**：同一网段扫描（如 `arp -a` 或 `nmap`）或路由器后台查看新 IP。
  2. **安装/升级过程中断或包损坏**：如 `apt`/`dpkg` 未完成（如 libc-bin、sgml-base 等配置失败），导致系统状态异常，甚至影响网络或启动。建议在能 SSH 时避免强制中断 `apt`/`dpkg`；若已断，下次上电后先 `sudo dpkg --configure -a`，必要时 `sudo apt --fix-broken install`。
  3. **修改了网络配置**：若改过 `/etc/network/interfaces` 或 Netplan/NetworkManager，配置错误会导致无 IP 或仅 link-local。
  4. **设备树/内核模块导致启动异常**：如 seeed-voicecard 的 overlay 或内核模块与当前内核不匹配，导致 I2C/I2S 或启动卡住，表现为上电后“无反应”或无法获取 IP。此时需**另一台电脑 + 读卡器**挂载 rootfs 或 boot 回退/删除 overlay 或相关配置。
  5. **电源/硬件**：供电不足或 SD 接触不良，导致反复重启或无法稳定启动，表现为“有时能连、有时不能”或完全无网。

### 2.2 避免再次失联的实践

- **在能 SSH 时先做**：
  - 若希望固定 IP：在树莓派上配置静态 IP（或路由器绑定 MAC），并记录。
  - 做一次 `ssh-copy-id`，后续用密钥登录，减少对密码的依赖。
- **做危险操作前**：
  - 修改 `/boot/firmware/config.txt`（或 `/boot/config.txt`）时，先备份；添加 `dtoverlay=seeed-8mic-voicecard` 等要确认型号（Pi 4 用标准 overlay 即可）。
  - 大批量 `apt install` 或 DKMS 编译时，不要中途断电或强制关 SSH；必要时用 `screen`/`tmux` 防止断线导致中断。
- **保留「救机」手段**：
  - 能用另一台电脑 + 读卡器时：拔卡挂载 boot/rootfs，恢复 config.txt、删除有问题的 overlay、或修复 `/etc/network` 配置。
  - 本仓库提供「SSH 直接可用固件」制作脚本，便于重新做一张干净、开箱即 SSH 的卡，再逐步装驱动和做 6 麦测试。

## 三、seeed-voicecard 与 6 麦测试

### 3.1 驱动安装（内核 6.12+）

- **官方 / HinTak 的 install.sh** 会安装 `raspberrypi-kernel-headers` 等，在 Debian Trixie / RPi OS 2025 上可能找不到该包，导致脚本报错退出。
- **处理**：修改 install.sh，在无法安装 `raspberrypi-kernel-headers` 时改为安装 `linux-headers-rpi-v8`（或 `linux-headers-$(uname -r)`），并仅在 `/lib/modules/$(uname -r)/build` 不存在时再要求「内核与 headers 一致」。
- **内核 API 变更**：6.12 中 `struct snd_soc_pcm_runtime` 的 `id` 成员被移除，直接使用 `rtd->id` 会编译失败。
- **处理**：在 `seeed-voicecard.c` 中为 6.12+ 用「dai_link 索引」替代 `rtd->id`（例如通过 `card->dai_link` 与 `rtd->dai_link` 的对应关系计算索引），并保持对旧内核的兼容（`#else` 分支仍用 `rtd->id`）。本仓库中 seeed-voicecard 已含该补丁。

### 3.2 6 麦设备与 overlay

- **Pi 4**：使用标准 `dtoverlay=seeed-8mic-voicecard` 即可；安装 seeed-voicecard 后，在 `config.txt` 中保留该 overlay，重启后 `arecord -L` 应能看到 seeed/ac108 相关设备。
- **Pi 5**：需使用仓库内提供的 Pi 5 专用 overlay（如 `seeed-8mic-voicecard-pi5`），且 6 通道在 Pi 5 上存在已知问题（多通道可能仅 1 路有数据），详见 `respeaker-6mic-docs/PI5_探索与结论.md`。

### 3.3 6 麦测试脚本

- 脚本位于 `respeaker-6mic-docs/scripts/`：`test_6mic.sh`、`verify_wav.py`。
- 依赖：树莓派上已安装 seeed-voicecard，且 `arecord -L` 中能看到 `seeed8micvoicec` 或 `ac108`。
- 若板子网络不稳，可在本机执行：将脚本 scp 到树莓派（如 `/tmp/respeaker_test/`），再 `ssh pi@<IP> 'cd /tmp/respeaker_test && ./test_6mic.sh'`；或先在本机下载好 seeed-voicecard 源码/包，再通过 scp 传到板子后编译安装，减少板子端下载时间。
